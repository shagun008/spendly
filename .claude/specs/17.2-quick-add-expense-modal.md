# Spec: Quick Add Expense Modal

## Roadmap description
Lets logged-in users add a new expense from the Profile page via a popup modal — no separate page, no nav item, same fast form.

## Overview
The standalone Add Expense page is one of three pages being consolidated into the Profile hub as part of feature 17. This release adds a "+" button next to the "My Profile" title heading that opens a modal containing the same form fields and validation as the current Add Expense page (amount, category, date, description). On submit the expense is saved and the page refreshes showing the new entry. After the modal is working, the standalone Add Expense nav item is removed from both desktop and mobile nav in `base.html`, the `templates/add_expense.html` template is deleted, the `/expenses/add` route is removed from `app.py`, and `static/css/add_expense.css` is deleted. This reduces navigation complexity and makes Profile the central hub for expense management.

## Error handling
- Invalid or missing form input (empty amount, negative amount, non-numeric amount, missing category, invalid date, description > 200 chars) — show inline error via flash message, modal stays open with form values preserved
- Unauthorised access — modal form is only shown to logged-in users; the add-expense backend handler must verify `session.get("user_id")` and redirect to `/login` if missing
- Database errors on insert — catch `psycopg2.Error`, rollback, flash "Could not save expense. Please try again.", modal stays open
- Submitting an empty form — amount and category are `required` attributes; browser validation catches this before submit

## UI/UX notes
- **"+" button placement:** To the right of the "My Profile" h1 title, on the same horizontal row, vertically centered. Use flexbox on `.profile-header-left` (already a flex container) or create a new wrapper that places the title and button side by side.
- **Modal behaviour:** Clicking "+" opens the modal with the same fields as the current Add Expense page (amount in ₹, category dropdown, date picker defaulting to today, optional description). Modal closes on successful submit, on clicking the X button, on clicking Cancel, on clicking the overlay backdrop, and on pressing Escape.
- **Form validation feedback:** Use flash messages displayed above the form inside the modal (or inline spans). On error, preserve the user's input values in the form fields.
- **Redirect behaviour:** On successful submit, redirect to `/profile` (or reload the page) so the new expense appears in the Recent Transactions table. Flash "Expense added." with success category.
- **Flash message copy:**
  - Success: "Expense added."
  - Amount error: "Amount must be a number greater than 0."
  - Category error: "Please select a valid category."
  - Date error: "Please enter a valid date."
  - Description error: "Description must be 200 characters or fewer."
  - Database error: "Could not save expense. Please try again."
- **Accessibility:** Modal must have `role="dialog"`, `aria-modal="true"`, and an `aria-labelledby` pointing to the title. Focus must be trapped inside the modal while open (first input receives focus on open, focus returns to the "+" button on close). Follow the same pattern as the existing Change Password modal in `profile.html`.
- **Responsive design:** Modal must be usable on mobile (max-width 600px breakpoint). Form inputs and buttons must be large enough to tap on mobile. Use existing CSS variables and breakpoints from `static/css/style.css`.

## Rules for implementation
- No SQLAlchemy or ORMs — use raw `sqlite3`/`psycopg2` queries only
- Parameterised queries only — never interpolate user data into SQL strings
- Use CSS variables — never hardcode hex values
- All templates extend `base.html`
- Auth guard on every protected route — check `session.get("user_id")` and redirect to `/login` if missing
- Ownership check on all expense mutations — queries must include `AND user_id = %s` to prevent cross-user access
- Flash messages for all user-facing outcomes — success and error states
- Currency displayed in ₹ — format amounts as `₹{amount:,.2f}`
- Dates stored as `YYYY-MM-DD` strings — validate with `datetime.strptime(value, "%Y-%m-%d")`
