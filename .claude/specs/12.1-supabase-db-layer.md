# Spec: Supabase DB Layer

## Overview
Spendly currently uses SQLite via the `sqlite3` standard library. This works locally but
is incompatible with Railway's ephemeral filesystem â€” every redeploy wipes the database.
This release replaces the entire database layer with psycopg2 connected to a Supabase
(managed PostgreSQL) instance. The swap covers the connection helper, schema creation,
seed data, and every parameterised query. No new routes or UI are introduced; the goal
is that all existing routes continue to behave identically, but now read from and write
to a persistent Postgres database. Success means the app boots cleanly, all routes pass
end-to-end, and data survives a server restart.

## Database changes
No new tables or columns. The existing schema is recreated in Postgres with these
DDL changes:
- `INTEGER PRIMARY KEY AUTOINCREMENT` â†’ `SERIAL PRIMARY KEY`
- `DEFAULT (datetime('now'))` â†’ `DEFAULT NOW()`
- `REFERENCES users(id)` â†’ `REFERENCES users(id) ON DELETE CASCADE`
- `UNIQUE (feature_id, user_id)` constraint syntax unchanged (Postgres compatible)
- `INSERT OR IGNORE` â†’ `INSERT â€¦ ON CONFLICT DO NOTHING`

The SQLite-specific `PRAGMA foreign_keys = ON` is removed (Postgres enforces FK
constraints by default).

## Error handling
- **Missing DATABASE_URL:** `get_db()` must raise a clear `RuntimeError` with a message
  explaining the env var is required â€” do not silently fall back to SQLite
- **Connection failure:** psycopg2 will raise `psycopg2.OperationalError`; let it
  propagate so Flask's error handler surfaces it during startup
- **Unique constraint violation on registration:** catch `psycopg2.errors.UniqueViolation`
  in `app.py` register route and flash "Email already registered."
- **`ON CONFLICT DO NOTHING` in feature_views and feature_votes:** these replace
  `INSERT OR IGNORE`; behaviour is identical â€” silently skip duplicate inserts

## Rules for implementation
- No SQLAlchemy or ORMs â€” use raw psycopg2 only
- Parameterised queries only â€” all user data via `%s` placeholders, never f-strings
- Use `RealDictCursor` on every connection so rows are accessed by column name,
  matching the existing `row["column"]` access pattern throughout queries.py
- DATABASE_URL must be read from the environment â€” never hardcode credentials
- All other CLAUDE.md rules (CSS variables, auth guards, ownership checks) are
  unchanged and still apply to the routes layer
