# Spec: Change Password

## Roadmap description
Lets logged-in users change their password securely by verifying their current password first.

## Overview
The profile page currently shows a disabled "Change Password" button in the user info card. This release wires up that button with a modal form and a backend route so authenticated users can change their password. The flow requires verifying the existing password (via `werkzeug.security.check_password_hash`) before accepting the new one, and enforces minimum length and confirmation matching. This is the first release of the User Profile Page Updates feature, establishing the pattern for profile-centric account management before later releases consolidate expense adding and analytics into the same page.

## Release context
Release 1 of 3 — MVP. The Change Password modal is the simplest self-contained piece of the User Profile Profile Page Updates thought and unblocks users who have no way to update their credentials today. Scope is strictly limited to the modal UI, a single POST route, and the password verification/update logic. Deferred to later releases: quick-add expense modal, embedded analytics dashboard, and removal of standalone Add Expense/Analytics pages. No open questions for this release.

## Depends on
No dependencies on other unreleased features. Builds on existing profile page (features 04–06), authentication (feature 03), and `werkzeug.security` helpers already in the codebase.

## Routes
- `POST /profile/change-password` — verifies current password, validates new password, updates the hash — access level: logged-in

## Database changes
No database changes. The `users` table already has a `password_hash` column (TEXT NOT NULL). The update rewrites this column for the authenticated user.

## Templates
- **Modify:** `templates/profile.html` — the disabled "Change Password" button (line 29) becomes an enabled button that opens a modal. Add a change-password modal form (current password, new password, confirm password) and the associated JavaScript to toggle visibility. Flash messages already flow through `base.html`'s flash block.

## Files to change

- `templates/profile.html` — replace disabled button with modal trigger, add modal markup, add JS for open/close/validation
- `static/css/profile.css` — add modal overlay, modal dialog, form field, and error styling using existing CSS variables
- `static/js/main.js` if needed — only if modal logic must live in the shared file; otherwise inline `<script>` in the profile template

## Files to create
No new files — the modal lives inline in `profile.html` and the route lives in the existing `app.py`.

## New dependencies
No new dependencies. Uses `werkzeug.security.generate_password_hash` (already a project dependency for registration) and `werkzeug.security.check_password_hash` (already imported in `app.py` for login).

## Error handling
- **Unauthorised access:** if `session.get("user_id")` is missing, redirect to `/login` (standard auth guard, consistent with all other protected routes)
- **Missing current password:** flash "Current password is required." and redirect to `/profile`
- **Wrong current password:** flash "Current password is incorrect." and redirect to `/profile`
- **Missing or too-short new password:** flash "New password must be at least 8 characters." (matches the 8-char minimum enforced at registration in `register`)
- **New password and confirmation mismatch:** flash "New passwords do not match." and redirect to `/profile`
- **Database error on UPDATE:** flash "Could not update password. Please try again." and redirect to `/profile`

## UI/UX notes
- The modal overlay should use a semi-transparent backdrop (`rgba(0,0,0,0.4)`) with a centered white card, matching the existing card style (`--paper-card`, `--border`, `--radius-md`)
- Modal has three fields: "Current password" (type password), "New password" (type password), "Confirm new password" (type password), each with `<label>` elements
- A close button (X icon via Lucide) in the top-right corner and a cancel button at the bottom dismiss the modal without submitting
- Change the existing disabled ghost button to an enabled `.btn-ghost` (remove `disabled` class and `btn-disabled` class); on hover it should match the existing ghost button hover style
- Flash messages follow the existing pattern: `flash(message)` for errors, `flash(message, "success")` for success
- After a successful password change, redirect to `/profile` with flash "Password changed.", "success"
- Responsive: modal should be full-width on mobile (with 16px side margins) and centered 440px wide on desktop, consistent with the `--auth-width` variable pattern
- All form fields must have associated `<label>` elements for accessibility; the modal must be dismissible via Escape key

## Rules for implementation
Read `CLAUDE.md` for the full list of project constraints.
If `CLAUDE.md` does not exist, apply these defaults:
- No SQLAlchemy or ORMs
- Parameterised queries only
- Passwords hashed with werkzeug
- Use CSS variables — never hardcode hex values
- All templates extend `base.html`

Additional project-specific rules:
- Import `generate_password_hash` from `werkzeug.security` in `app.py` (only `check_password_hash` is imported today)
- Use a parameterised `UPDATE users SET password_hash = %s WHERE id = %s` query
- The route must include `AND user_id = ?` ownership (trivially satisfied since `user_id` comes from `session`, but use it anyway for consistency)
- Modal state must be client-side only — no server-side rendering of modal errors; all validation feedback comes from flash messages after redirect
