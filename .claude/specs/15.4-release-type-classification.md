# Spec: Release-Level Type Classification

## Roadmap description
Adds a type badge (New Feature, Enhancement, or Bug Fix) to each release row on the roadmap, so visitors can see at a glance what kind of work each release represents.

## Overview
Currently the `type` column on the `features` table is set on parent feature rows during `/capture-thoughts` and hardcoded to `'feature'`, which is meaningless. The meaningful classification — whether a release adds something new, improves something existing, or fixes a bug — is only knowable at planning time, not capture time. This release moves type to release sub-rows: parent rows get NULL (they are containers only), and each release sub-row gets one of `new-feature`, `enhancement`, or `bug-fix` set during `/plan-release`. The roadmap page renders a small type badge on each release sub-row so visitors can see the nature of the work at a glance.

## Release context
Release 4 of 5 — Developer Roadmap Page.

- `roadmap.css` — add badge styles for the three type values.

## UI/UX notes
- Type badge renders on release sub-rows only (`.roadmap-row--release`), not on parent feature rows.
- Badge sits inline after the release title, before the status badge.
- Three styles needed: `new-feature` (blue tint), `enhancement` (amber tint — use `--accent-2-light` / `--accent-2`), `bug-fix` (danger tint — use `--danger-light` / `--danger`).
- Badge text should be human-readable: `New Feature`, `Enhancement`, `Bug Fix`.
- Badge must not wrap — `white-space: nowrap`.
- No new breakpoints needed — badge is small enough to fit at mobile widths.
