# Spec: Profile Card Layout & Dropdown Updates

## Roadmap description
Gives the navbar dropdown distinct icons and a Change Password option, and packs the profile card with useful spending stats.

## Overview
The current profile experience has three rough edges that confuse logged-in users and waste space. First, the navbar dropdown trigger and the "My Profile" menu item use the same `user` Lucide icon, making them visually indistinguishable. Second, the only way to reach the (already-built) Change Password flow is a button sitting on a mostly-empty profile card — the navbar dropdown exposes "My Profile" and "Log Out" but not "Change Password". Third, the profile card shows only the user's name, email, and member-since line, leaving the rest of the card empty, while the three key stats (Total Spent This Month, Transactions, Top Category) sit in a separate `stats-row` further down the page. This release consolidates those issues: distinct icons for the dropdown trigger vs the "My Profile" item, a "Change Password" row inserted between "My Profile" and "Log Out", and a compact 4-row profile card that stacks the user-info row plus three stat rows inside the card's current height — removing both the redundant Change Password button and the standalone `stats-row`. All three changes are UI-only: no new routes, no DB schema changes, no new dependencies.

## Release context
Release 1 of 1 — MVP. Feature 20 was decomposed into a single release because all three changes are tightly coupled UI work on the same two surfaces (navbar + profile card). Nothing is deferred. No open questions remain from the processed thought or release plan.

## Depends on
- 19.2-navbar-user-menu — the navbar user dropdown must already exist (it does, in `base.html`).
- 17.1-change-password — the Change Password modal + route must already exist so the dropdown row has somewhere to point.

## Routes
No new routes. The "Change Password" dropdown row links to the existing `/profile/change-password` action via the existing modal; the open/close trigger moves from the profile card button to the dropdown row.

## Database changes
No database changes. The three stat rows (`total_spent`, `transaction_count`, `top_category`) are already produced by `get_summary_stats()` in `database/queries.py`. The stats-row values are moved verbatim into the profile card; no new columns or queries are needed.

## Templates
- **Modify:**
  - `templates/base.html` — give the dropdown trigger a distinct icon (e.g. `chevron-down` next to / instead of `user`), and change the "My Profile" item icon to something distinct from the trigger (e.g. `user-circle`). Insert a "Change Password" dropdown item between "My Profile" and "Log Out" that opens the existing `#change-password-modal`.
  - `templates/profile.html` — merge the standalone `stats-row` into the `profile-card` as compact stat rows (Total Spent This Month, Transactions, Top Category) stacked beneath the user-info row. Remove the old "Change password" button from `profile-card-actions`. Remove the now-duplicate `stats-row` block.

## Files to change
- `templates/base.html` — navbar dropdown icons + new Change Password row.
- `templates/profile.html` — 4-row profile card layout, remove old button and duplicate stats-row, remove the card's inline change-password JS now that the trigger moved to the navbar.
- `static/css/profile.css` — new styles for the in-card stat rows; remove/adjust styles for `.stats-row` / `.stat-card` if they become unused, and adjust `.profile-card` to host the stacked layout. Keep responsive behaviour (mobile stacked).
- `static/css/style.css` — adjust `.nav-user-trigger` icon styling if the trigger icon changes.

## Files to create
None.

## New dependencies
No new dependencies. All icons must come from the existing Lucide CDN (`user-circle`, `chevron-down`, `lock`, `log-out`, plus existing `credit-card`, `list`, `tag` for the stat rows).

## Error handling
- This feature is UI-only and adds no new user input paths, so there are no new validation or DB-error cases.
- The Change Password flow itself is unchanged; its existing error handling (wrong current password, mismatch on confirmation) stays as-is in the `change_password` route.
- Graceful degradation: if `stats.transaction_count` is 0, the stat rows should still render — `total_spent` shows `₹0.00` and `top_category` shows `—` (already the behaviour of `get_summary_stats`).

## UI/UX notes
- **Icons:** The trigger and the "My Profile" item must not share the same icon. Trigger: `chevron-down` (or `user` + `chevron-down`). "My Profile" item: `user-circle`. "Change Password" item: `lock`. Keep `log-out` for Log Out.
- **Dropdown order:** My Profile → Change Password → Log Out.
- **Profile card layout:** Inside `.profile-card`, stack four rows: (1) avatar + name/email/member-since, (2) Total Spent This Month, (3) Transactions, (4) Top Category. All four must fit within the current card height. Use a compact row style (icon + label + value), not the large standalone stat cards.
- **Change Password trigger:** The modal open/close logic currently lives in `profile.html` keyed off `#change-password-btn`. Move the trigger to the new dropdown row; keep the modal markup and its JS in `profile.html`.
- **Accessibility:** all dropdown items keep `role="menuitem"`; new items get the same focus styles as existing ones. Stat rows are presentational — no interactive elements to label. colour contrast must meet WCAG AA; keep using CSS variables.
- **Responsive:** on mobile (≤600px) the card rows may collapse to a tighter stack. Ensure the 4-row card does not overflow its container on narrow screens. Keep using existing breakpoints from `static/css/style.css`.
- **Flash messages:** none added (no new mutations).

## Rules for implementation
- No SQLAlchemy or ORMs.
- Parameterised queries only (no DB changes here, but any reuse of existing queries must stay parameterised).
- Use CSS variables — never hardcode hex values.
- All templates extend `base.html`.
- Reuse existing Lucide icons only — do not add a new icon library or CDN.
- Mobile responsiveness must be preserved using existing breakpoints.
- Do not modify `app.py`, `database/queries.py`, or `database/db.py` — this is a UI-only change.
