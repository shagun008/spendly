# Spec: Local Data Migration to Supabase

## Overview
With the database layer now running on Supabase (Release 12.1), any existing local
data — users and expenses accumulated during development in `spendly.db` — is
stranded in SQLite. This release provides a one-off migration script that exports
that local data to CSV files, imports them into Supabase preserving original IDs,
and resets Postgres sequences so future inserts don't collide with migrated IDs.
The `spendly.db` file is kept intact throughout as a backup; if anything goes wrong
the script can simply be re-run after cleaning up Supabase. Success means every
user and expense that existed locally appears in Supabase with the same IDs and
data intact.

## Release context
Release 2 of 2 — Local Data Migration.

**In scope for this release:**
- Script to connect to local `spendly.db` and export `users` and `expenses` tables to CSV
- Script to import those CSVs into Supabase via psycopg2, inserting rows with their
  original `id` values (bypassing SERIAL auto-increment)
- After import: reset Postgres sequences (`users_id_seq`, `expenses_id_seq`) to
  `MAX(id)` so subsequent inserts don't collide
- Row-count and spot-check verification printed to stdout after import
- `spendly.db` is never deleted or modified — it remains as a safe backup

**Deferred / out of scope:**
- Migrating `feature_requests`, `feature_votes`, `feature_views` — these tables were
  introduced alongside Supabase (11.x) and have no local SQLite history
- Railway SQLite data — the Railway SQLite DB will be wiped and will start with the new Supabase datastore; only local dev data is in scope to be migrated here
- Dev/prod Supabase split — not needed until production has real user data

**Open questions:** None remaining.

## Script: `scripts/migrate_local_to_supabase.py`

The script must:

1. **Check prerequisites** — confirm `spendly.db` exists in the project root;
   confirm `DATABASE_URL` is set in the environment (load `.env` with `python-dotenv`);
   abort with a clear error message if either is missing.

2. **Export from SQLite** — open `spendly.db`, query `SELECT * FROM users` and
   `SELECT * FROM expenses`, write each to a temporary CSV in memory (no files written
   to disk). Print row counts before proceeding.

3. **Import into Supabase** — open a psycopg2 connection to `DATABASE_URL`.
   For `users`: `INSERT INTO users (id, name, email, password_hash, created_at) VALUES (%s, %s, %s, %s, %s) ON CONFLICT (id) DO NOTHING`.
   For `expenses`: `INSERT INTO expenses (id, user_id, amount, category, date, description, created_at) VALUES (%s, %s, %s, %s, %s, %s, %s) ON CONFLICT (id) DO NOTHING`.
   `ON CONFLICT DO NOTHING` makes the script safe to re-run; rows already migrated
   are skipped without error.

4. **Reset sequences** — after import, run:
   ```sql
   SELECT setval('users_id_seq', COALESCE(MAX(id), 1)) FROM users;
   SELECT setval('expenses_id_seq', COALESCE(MAX(id), 1)) FROM expenses;
   ```
   Print the new sequence values to stdout.

5. **Verify** — query row counts from Supabase for both tables and print a
   comparison table:
   ```
   Table      SQLite rows   Supabase rows   Match?
   users      X             X               ✓
   expenses   X             X               ✓
   ```
   If counts don't match, print a warning but do not abort (re-runnable).

6. **Commit and close** — commit the transaction; close both connections.

## Error handling
- **Missing `spendly.db`:** abort with `FileNotFoundError` and a message explaining
  where the file is expected (`<project_root>/spendly.db`)
- **Missing `DATABASE_URL`:** abort with `RuntimeError` matching the message in
  `database/db.py`
- **psycopg2 connection failure:** let `OperationalError` propagate — the traceback
  is sufficient for a dev-facing script
- **Partial import failure:** any insert error inside the loop should roll back the
  transaction and print which table and row caused the failure, so the user can
  investigate before re-running
- **Sequences already higher than migrated data:** `setval` with the MAX-based value
  is safe; if the sequence is already higher, the `COALESCE(..., 1)` guards against
  an empty table edge case

## UI/UX notes
This is a developer-facing CLI script — no UI. Output should be clear and readable
in a terminal. Use print statements (not logging) so output is immediately visible.
No Flask app context is needed; the script runs standalone.

## Rules for implementation
- Parameterised queries only — no f-strings or string interpolation into SQL
- Load `.env` via `python-dotenv` so the script works in the same local dev environment
  as the app without needing environment variables to be manually exported
- Script must be idempotent — safe to run multiple times without duplicating data
  (`ON CONFLICT DO NOTHING` guarantees this)
- Do not delete or modify `spendly.db` at any point
- No SQLAlchemy or ORMs
