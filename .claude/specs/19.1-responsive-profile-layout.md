# Spec: Responsive Profile Layout

## Roadmap description
Makes the Profile Card and Analytics Dashboard sit side-by-side on desktop screens, stacking vertically on mobile for a better use of space.

## Overview
The current Profile page renders the user info card and the analytics dashboard as separate full-width sections stacked vertically, leaving wide screens underutilised. This release wraps those two sections in a responsive two-column container — Profile Card (left, narrower) and Analytics Dashboard (right, wider) — displayed side-by-side on desktop (above the 900px breakpoint) and stacked on smaller screens. The change is purely structural (a wrapper div in the template and CSS flexbox rules). No routes, database queries, or JavaScript are modified.

## Release context
Release 1 of 2 — Responsive Profile Layout (MVP). This release covers the CSS-only side-by-side layout for the Profile Card and Analytics Dashboard on desktop screens. Release 2 (Navbar User Menu) will modernise the navbar with a user icon and dropdown but is fully independent of this work. No open questions remain for this release.

## UI/UX notes
- The Profile Card should occupy roughly 1/3 of the width on desktop, and the Analytics Dashboard roughly 2/3, matching the visual weight of their content
- The existing 900px and 768px breakpoints in `profile.css` already handle the stacked layout on smaller screens — the new `.profile-top-row` must collapse to `flex-direction: column` at the same breakpoint
- On mobile (≤768px), the Profile Card and Analytics Dashboard must stack in the same vertical order as today (Profile Card first, then Analytics Dashboard)
- No visual regressions: spacing, padding, font sizes, and card styles must remain identical to the current layout when stacked
- Accessibility: the layout change is purely visual — no focus order or ARIA attribute changes needed

## Rules for implementation
- No SQLAlchemy or ORMs — use raw SQL queries only
- Parameterised queries only — never interpolate user data into SQL strings
- Use CSS variables — never hardcode hex values
- All templates extend `base.html`
- No new routes or backend changes
