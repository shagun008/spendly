# Spec: Release Notes Modal

## Roadmap description
Click the green Tested or Reviewed dot on any release to see the full test report or code review report — an instant audit trail.

## Overview

The roadmap page (Release 15.1–15.4) currently shows a pipeline table with green dots for completed stages, but clicking a dot does nothing. The processed thought for this release identifies the problem: visitors can see *when* a stage was completed but not *what was found or done*. This release makes **only the Tested and Reviewed stage dots** on release sub-rows clickable. Clicking the Tested dot opens a modal showing the test report; clicking the Reviewed dot opens a modal showing the code review report. All other stage dots (Captured, Planned, Spec'd, Implemented, Shipped) remain display-only — no click interaction, no cursor change.

Two new `TEXT` columns are added to the `features` table (`test_report`, `review_report`). The existing `/test-feature` and `/code-review-feature` harness commands are extended to write their reports into these columns as part of their stamp step. The modal is dismissible via click-outside or Escape key. A Tested or Reviewed dot is only clickable when its respective report exists. Parent feature row dots never trigger the modal — only release sub-row dots do.

## Error handling

- **No report exists for a stage:** that stage's dot is not clickable — no cursor change, no hover effect, same visual treatment as before this release
- **Tested dot clicked:** modal shows the test report for that release
- **Reviewed dot clicked:** modal shows the code review report for that release
- **Modal already open and user clicks another dot:** close the current modal, open the new one
- **Click-outside dismiss:** clicking anywhere outside the modal content closes it
- **Escape key dismiss:** pressing Escape closes the modal
- **DB write failure in harness commands:** the command prints an error but does not crash; the stamp step continues

## UI/UX notes

- Only Tested and Reviewed dots get `cursor: pointer` and a hover effect when their respective report exists
- All other dots (Captured, Planned, Spec'd, Implemented, Shipped, and Tested/Reviewed with no report) keep `cursor: default` — unchanged from before this release
- Modal overlay: semi-transparent dark background covering the full viewport
- Modal card: centered, white background, max-width ~600px, max-height 80vh with internal scroll
- Modal title: `"<number> — <title>"` (e.g. "15.4 — Release-Level Type Classification")
- Summary section at top: key metrics (pass/fail count, issue count, overall verdict) extracted from the reports
- Reports displayed in sections with clear headings: "Test Report" and "Code Review Report"
- Close button (X) in the top-right corner of the modal
- All text uses existing CSS variables; modal uses `--paper-card`, `--ink`, `--border`, `--radius-md`
- Responsive: modal takes full width minus 1rem padding on mobile (`max-width: calc(100vw - 2rem)`)
- Accessibility: modal has `role="dialog"` and `aria-modal="true"`; focus trapped inside while open; Escape to dismiss

## Rules for implementation

Read `CLAUDE.md` for the full list of project constraints. Key reminders:

- `ALTER TABLE ADD COLUMN IF NOT EXISTS` for idempotent column additions
