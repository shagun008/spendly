# Spec: Harness Integration (Live Updates)

## Roadmap description
Keeps the public roadmap up to date automatically as features move through the pipeline тАФ no manual editing needed.

## Overview
The `/roadmap` page (built in 15.1) reads from the `features` DB table, but that table is only populated via `seed_features()` and a few manual `UPDATE` calls. Right now, when a feature advances through the pipeline the DB is not written to automatically тАФ the table is stale the moment any harness command runs. This release wires every harness command to the DB so the roadmap stays live across all six pipeline stages: `/capture-thoughts` inserts new rows and sets `captured_at`; `/plan-release` sets `planned_at` and inserts release sub-rows; `/create-spec` sets `spec_at`; `/implement-feature` sets `implemented_at`; `/test-feature` sets `tested_at`; `/code-review-feature` sets `reviewed_at`; `/ship-feature` sets `shipped_at`; `/deploy` sets `deployed_at`. The `/status` command is updated to read from the DB instead of regenerating from `registry.md`. `status.md` is removed тАФ the DB is the sole live source of truth.

## Release context

**In scope:**
- `/capture-thoughts`: after writing the processed-thought file and updating `registry.md`, INSERT a new row into the `features` table and set `captured_at = NOW()`. Use `INSERT ... ON CONFLICT (number) DO UPDATE SET captured_at = NOW()` to make it idempotent.
- `/plan-release`: after writing the release plan file and updating `registry.md`, set `planned_at = NOW()` on the parent feature row, and INSERT one release sub-row per release defined in the plan (each with `planned_at = NOW()`). - `/create-spec`: after saving the spec file and updating `registry.md`, set `spec_at = NOW()` on the specific release row (e.g. `15.3`).
- `/implement-feature`: after completing implementation, set `implemented_at = NOW()` on the specific release row.
- `/test-feature`: after tests pass, set `tested_at = NOW()` on the specific release row.
- `/code-review-feature`: after the review completes, set `reviewed_at = NOW()` on the specific release row.
- `/ship-feature`: after merging, set `shipped_at = NOW()` on the release row. Also check if all release rows for the parent feature are now shipped тАФ if so, stamp `shipped_at` on the parent row too. (Step 9a in the current command already does part of this; extend it to cover the full logic.)

## Error handling
- All DB writes in harness commands must be wrapped in a try/except. If the DB write fails, log the error to the console and continue тАФ do not block the command from completing its file and registry updates.
- INSERT statements must use `ON CONFLICT (number) DO UPDATE` or `ON CONFLICT (number) DO NOTHING` as appropriate so re-running a command never raises a duplicate-key error.
- `/status` must handle the case where the `features` table is empty тАФ render an empty grouped output with a note "No features in DB yet."
- If `DATABASE_URL` is not set in the environment, log a warning and skip the DB write gracefully.
- `/deploy` DB stamp: if no row has `shipped_at IS NOT NULL AND deployed_at IS NULL`, log "No undeployed release found тАФ skipping DB stamp" and continue.
