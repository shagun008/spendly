# Spec: Roadmap Pipeline

## Overview
Visitors and users currently have no visibility into what is being built on Spendly. This release introduces a public `/roadmap` page that renders a pipeline table — one row per feature, one column per dev stage (Captured → Planned → In Progress → In Review → Shipped) — with a tick and timestamp for each completed stage. This is the MVP release: it creates the `features` DB table, seeds it from the existing registry (backfilling existing shipped features with "May 2026" timestamps), adds the `GET /roadmap` route, adds a nav link visible when logged out, and creates `roadmap.css`. It is the foundation that Releases 2 and 3 build on.

## Release context

## Database changes
New table `features`:

| Column          | Type      | Constraints                                  |
|-----------------|-----------|----------------------------------------------|
| id              | SERIAL    | PRIMARY KEY                                  |
| number          | TEXT      | UNIQUE NOT NULL (e.g. "15.1", "11", "12.2")  |
| parent_number   | TEXT      | nullable (e.g. "15" for release "15.1")      |
| title           | TEXT      | NOT NULL                                     |
| slug            | TEXT      | NOT NULL                                     |
| type            | TEXT      | NOT NULL DEFAULT 'feature'                   |
| description     | TEXT      | nullable                                     |
| captured_at     | TIMESTAMP | nullable                                     |
| planned_at      | TIMESTAMP | nullable                                     |
| in_progress_at  | TIMESTAMP | nullable                                     |
| in_review_at    | TIMESTAMP | nullable                                     |
| shipped_at      | TIMESTAMP | nullable                                     |
| created_at      | TIMESTAMP | DEFAULT NOW()                                |

No changes to existing tables (`users`, `expenses`, `feature_requests`, `feature_votes`, `feature_views`).

## Error handling
- If the `features` table is empty (e.g. seed has not run), the `/roadmap` page renders with an empty table and a "No features yet" message — no 500 error.
- `seed_features()` must be idempotent: check if any rows already exist before inserting; skip if data is present.
- DB connection errors bubble up as 500 (no special handling needed beyond Flask's default).

## UI/UX notes
- **Pipeline table layout:** Each row = one feature. Columns: Feature #, Title, Captured, Planned, In Progress, In Review, Shipped. Stage columns show a ✓ glyph (or Lucide `check` icon) + the month/year (e.g. "May 2026") when the stage is complete; blank when not yet reached.
- **Status badge:** Derive current status from the rightmost non-null stage column. Display as a small badge beside the title (e.g. "Shipped", "In Progress").
- **Nav link:** Add "Roadmap" to the nav in `base.html`, positioned after the existing public links. Must be visible when logged out.
- **No flash messages** — this is a read-only public page.
- **Responsive:** Table scrolls horizontally on mobile (`overflow-x: auto` on a wrapper div). Use existing breakpoints (`@media (max-width: 900px)`, `@media (max-width: 600px)`).
- **Colour:** Use CSS variables only — `--accent` for the ✓ glyphs, `--ink-muted` for empty stage cells, `--paper-card` for the table background.
- **Accessibility:** Table must have `<th scope="col">` headers. Status badges must have sufficient contrast (WCAG AA).
- **Page title:** "Roadmap — Spendly"

## Rules for implementation
Follow all rules in `CLAUDE.md`:
- No SQLAlchemy or ORMs — raw `psycopg2` queries only
- Parameterised queries only
- Use CSS variables — never hardcode hex values
- All templates extend `base.html`
- No auth guard on `/roadmap` — it is intentionally public
