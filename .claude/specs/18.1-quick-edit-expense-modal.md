# Spec: Quick Edit Expense Modal

## Roadmap description
Edit any expense straight from your Profile page in a quick modal — no more navigating to a separate edit page.

## Overview
The dedicated Edit Expense page (`/expenses/<id>/edit`) is the last of the standalone expense pages still standing after the Quick Add Expense Modal (17.2) consolidated Add Expense into the Profile hub. This release replaces the separate Edit Expense page with a modal on the Profile page, using the same fields, validation, and ownership guarantees the standalone page has today. Clicking "Edit" on an expense row opens a modal pre-populated with that expense's current values; submitting saves the change in place via the existing `update_expense` query helper and refreshes the Profile page. After the modal works, the standalone edit page, its route, its CSS, and any unused code are deleted. This keeps users in the flow of reviewing their expenses and completes the consolidation of expense management into the Profile hub.

## Error handling
- Invalid or missing form input (empty amount, negative/zero amount, non-numeric amount, missing category, invalid date, description > 200 chars) — flash an error message, modal stays open with the submitted values preserved.
- Unauthorised access — the edit-expense backend handler must verify `session.get("user_id")` and redirect to `/login` if missing.
- Editing another user's expense — `update_expense` already scopes to `user_id`; if 0 rows are affected, flash "Expense not found." and redirect to `/profile`.
- Non-existent expense id — flash "Expense not found." and redirect to `/profile`.
- Database errors on update — catch `psycopg2.Error`, rollback, flash "Could not update expense. Please try again.", modal stays open.
- Submitting an empty form — amount and category are `required` attributes; browser validation catches this before submit.

## UI/UX notes
- **Edit button placement:** Replace the current text "Edit" link in the Actions cell of each transaction row with a button (e.g. `<button class="tx-edit-btn" data-expense='{{ tx | tojson }}'>Edit</button>`). Style consistently with the existing `.tx-delete-btn`.
- **Modal behaviour:** Clicking Edit on a row opens the edit modal pre-populated with that expense's current values (amount, category selected, date, description). Modal closes on successful submit, on clicking the X button, on clicking Cancel, on clicking the overlay backdrop, and on pressing Escape.
- **Form validation feedback:** Use flash messages displayed above the form inside the modal. On error, preserve the user's submitted input values in the form fields.
- **Redirect behaviour:** On successful submit, redirect to `/profile` so the updated values appear in the transactions table. Flash "Expense updated." with success category.
- **Flash message copy:**
  - Success: "Expense updated."
  - Amount error: "Amount must be a number greater than 0."
  - Category error: "Please select a valid category."
  - Date error: "Please enter a valid date."
  - Description error: "Description must be 200 characters or fewer."
  - Database error: "Could not update expense. Please try again."
  - Not found: "Expense not found."
- **Accessibility:** Modal must have `role="dialog"`, `aria-modal="true"`, and an `aria-labelledby` pointing to the title. Focus must be trapped inside the modal while open (first input receives focus on open, focus returns to the Edit button on close). Follow the same pattern as the existing Quick Add Expense and Change Password modals in `profile.html`.
- **Responsive design:** Modal must be usable on mobile (max-width 600px breakpoint). Form inputs and buttons must be large enough to tap on mobile. Use existing CSS variables and breakpoints from `static/css/style.css`.

## Rules for implementation
- No SQLAlchemy or ORMs — use raw `psycopg2` queries only
- Parameterised queries only — never interpolate user data into SQL strings
- Use CSS variables — never hardcode hex values
- All templates extend `base.html`
- Auth guard on every protected route — check `session.get("user_id")` and redirect to `/login` if missing
- Ownership check on all expense mutations — queries must include `AND user_id = %s` to prevent cross-user access (already enforced in `update_expense`)
- Flash messages for all user-facing outcomes — success and error states
- Currency displayed in ₹ — format amounts as `₹{amount:,.2f}`
- Dates stored as `YYYY-MM-DD` strings — validate with `datetime.strptime(value, "%Y-%m-%d")`
