# Spec: Improvement Loop Skill

## Roadmap description
A structured improvement cycle that catches test failures and code review issues, fixes root causes, and prevents the same problems from recurring.

## Overview
When tests fail or code review finds issues, developers currently make ad-hoc fixes with no structured process — leading to recurring problems and gradual quality degradation. The Post-Review Improvement Loop is a manually-triggered slash command (`/improvement-loop <feature_number>`) that the developer runs after `/test-feature` or `/code-review-feature` completes with failures. It follows a disciplined 5-phase workflow — Gather Context, Root Cause Analysis, Architectural Assessment, Generate Improvement Plan, and Implementation + Validation + Learning Capture — to ensure every fix addresses the root cause and leaves the codebase healthier than before. A decision framework (Option A/B/C) ensures the smallest sufficient change is made. Learning captures are logged to `.claude/features/improvements/` for future reference.

## Error handling
- If the feature number argument is missing, stop and print usage: `/improvement-loop <feature_number>`
- If no spec file exists for the feature number, warn and continue — the loop can still gather context from source files
- If no failures are found (tests already pass, no review issues), report "No issues detected — nothing to improve" and exit
- If the improvement loop cannot resolve the issue after analysis, report the findings and recommend manual intervention
- All file writes (learning capture) are best-effort — if the write fails, log the error and continue

## UI/UX notes
No user-facing UI. This is a CLI slash command that produces terminal output only.

Output format for each phase:
```
Phase 1 — Gather Context
  <summary of what was collected>

Phase 2 — Root Cause Analysis
  <root cause identified>

Phase 3 — Architectural Assessment
  <architectural health check>

Phase 4 — Improvement Plan
  <chosen option (A/B/C) with rationale>

Phase 5 — Implementation + Validation + Learning Capture
  <what was changed, test results, learning logged>
```

Final output must answer: "Is the system better than it was before the issue was discovered?" — if not clearly "yes", the loop should recommend further action.
