Track feature request views per session, not per unique user.

Current behaviour: the feature_views table has a unique constraint on
(feature_id, viewer_id), so a user can only ever be counted once per feature
regardless of how many times they return.

Desired behaviour: count one view per user per session per feature.
- If a user views a feature card and then clicks it again in the same session,
  the view count does not increment.
- If the user logs out and logs back in (new session), viewing the same feature
  again counts as a new view.

Implementation idea:
- Use Flask's session object to track which feature ids have been viewed in
  the current session (e.g. session['viewed_features'] as a list of ids)
- Before inserting into feature_views and incrementing the views counter on
  feature_requests, check if the feature_id is already in session['viewed_features']
- If yes: skip — no insert, no increment
- If no: insert a new row into feature_views, increment views on feature_requests,
  add feature_id to session['viewed_features']
- Drop the unique constraint on feature_views(feature_id, viewer_id) since the
  same user can now have multiple rows across different sessions

This approach requires no timestamp comparisons and no additional database queries
for the cooldown check — the session object handles it entirely.
