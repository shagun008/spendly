--- HARNESS SECTION ---

The harness is a structured set of Claude Code slash commands that automate
every stage of feature development. Name each command explicitly in the README:

  /capture-thoughts   — reads free-form notes from user-thoughts/, summarises
                        them into processed-thoughts/
  /plan-release       — decomposes a feature into release-sized units, writes
                        a release plan file
  /create-spec        — writes a formal spec file and creates a feature branch
  /test-feature       — generates pytest tests from the spec, then runs them
  /code-review-feature — launches a security reviewer and quality reviewer in
                        parallel (two specialised agents running simultaneously)
  /ship-feature       — commits, opens a PR, squash-merges, and cleans up the
                        branch

Each feature moves through a tracked status lifecycle:

  💡 Captured → 📋 Planned → 🔧 In Progress → 👀 In Review → ✅ Shipped

Workflow diagram (ASCII — suitable for README code block):

  User notes / screenshot
          │
          ▼
  /capture-thoughts          (💡 Captured)
          │
          ▼
  /plan-release              (📋 Planned)
          │
          ▼
  /create-spec + Plan Mode   (🔧 In Progress)
          │
          ▼
  /test-feature              (👀 In Review)
          │
          ▼
  /code-review-feature       (security + quality agents, parallel)
          │
          ▼
  /ship-feature              (✅ Shipped)

Evolution note worth including: the harness was capable enough to manage a
non-trivial database migration (SQLite → Supabase PostgreSQL) as two spec'd
features (12.1 and 12.2), including data migration scripts — not just schema
changes. For a CFO or COO audience, this is a cost and risk reduction story:
infrastructure changes handled through the same governed workflow as features,
not as manual one-offs.


--- APP SECTION ---

App name: Spendly

Problem it solves: individuals who want a simple, private way to log and review
personal expenses without relying on spreadsheets or complex financial tools.
Built specifically for Indian users — all amounts displayed in ₹.

Features shipped (12 features, 15 spec releases):

  Auth          — register, login, logout with hashed passwords (werkzeug)
  Profile       — expense list with date-range filters and running totals
  Expenses      — add, edit, delete expenses across 7 categories
                  (Food, Transport, Bills, Health, Entertainment, Shopping, Other)
  Analytics     — visual breakdown of spending by category and over time
  Feature Requests — community board where users submit, upvote, and discover
                  feature ideas; trending sorted by votes and unique views
  Mobile Nav    — hamburger navigation, fully responsive at 600px breakpoint

Tech stack (for the README tech table):

  Backend       Flask 3.x (Python 3)
  Database      PostgreSQL via Supabase (psycopg2-binary)
  Auth          werkzeug.security
  Frontend      Jinja2 templates, vanilla CSS, vanilla JS
  Icons         Lucide (CDN)
  Fonts         DM Serif Display + DM Sans (Google Fonts)
  Testing       pytest + pytest-flask (one test file per feature)
  Server        gunicorn (Procfile)

Stat line for "Why This Matters" section (tight, scannable):
  12 features · 15 spec releases · 0 unreviewed merges · 1 DB migration
  handled as a governed harness feature

TONE NOTES
================================================================================

- The README is a professional showcase, not a setup guide or contribution
  invite — write every line with that frame in mind
- Lead with strategic outcomes: "a repeatable, governed AI delivery pipeline"
  is the message; Spendly is the evidence
- "Why This Matters" is the most important section — it must work as a
  standalone paragraph for a CEO or CDO who reads nothing else
- For CFO/COO readers: the DB migration story is about risk reduction and
  process discipline, not technical architecture — name it that way
- For CAIO/CDO readers: the harness demonstrates structured AI integration
  into a delivery workflow, not casual AI use — the governance (spec, test,
  parallel review, tracked status) is the differentiator
- Quantify wherever possible: "12 features", "15 releases", "0 unreviewed
  merges" — concrete numbers land harder than adjectives
- The harness narrative is methodology-first; commands are supporting evidence
  for technical readers who want to verify the claims
- Avoid imperative syntax in narrative sections — descriptive language only
  ("the pipeline enforces a review before any merge" not "run /ship-feature")
- Screenshots and live link near the top — visual credibility for GTM,
  Forward Deployment, and business readers before they read a word
- Technical depth is fully present — positioned in sections 5 and 6, after
  the value proposition is established
- No superlatives, no marketing language — let the stat line and feature
  count do the work
