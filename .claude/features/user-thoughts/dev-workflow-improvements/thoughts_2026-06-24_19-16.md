# Auto-Execute Mode for /test-feature and /code-review-feature

## Problem
The current `/test-feature` and `/code-review-feature` commands produce reports (test failures, review findings) but then ask the user for approval before fixing anything. This breaks flow — the user has to re-read the report, decide what to fix, and often says "fix everything anyway." The approval step adds friction without adding safety, because the findings are already presented and the user can see exactly what will change.

## Proposal
Add a `--auto` flag to `/test-feature` and `/code-review-feature` that controls whether the action plan is auto-executed:

- **Without `--auto` (default)**: existing behavior — present report, ask user "Do you want me to implement fixes?", wait for approval
- **With `--auto`**: present report → stamp to DB → auto-execute all fixes → show summary of what was changed

The flag is a CLI argument at invocation time, not a persistent config file. The user decides per-command whether to auto-execute.

Examples:
- `/code-review-feature 18.1-quick-edit-expense-modal --auto` → review, stamp, auto-fix
- `/code-review-feature 18.1-quick-edit-expense-modal` → review, stamp, ask user
- `/test-feature 18.1-quick-edit-expense-modal --auto` → write, run, stamp, auto-fix failures
- `/test-feature 18.1-quick-edit-expense-modal` → write, run, stamp only if pass

## What changes

### `/code-review-feature`
**Current flow:**
1. Parallel review
2. Present unified report
3. *Ask user for approval* ← removed in auto mode
4. Implement fixes
5. Update status

**Auto mode flow:**
1. Parallel review
2. Present unified report
3. Stamp review report to DB (always)
4. Auto-execute combined action plan (no prompt)
5. Show summary: "Fixed X items. Re-running tests."
6. Re-run tests, regenerate review report, stamp updated version
7. Update status

**Manual mode flow (default / flag off):**
- Same as current: present → ask → implement only if user says yes

### `/test-feature`
**Current flow:**
1. Write tests
2. Run tests
3. Present pass/fail report
4. *Stamped in DB only if tests pass*
5. No fixes — user has to improvise

**Auto mode flow:**
1. Write tests
2. Run tests
3. Present pass/fail report
4. Stamp test report to DB (always, even on failure)
5. For each failing test: classify as bug vs wrong test
6. Auto-fix bugs in implementation
7. For tests that are wrong because the spec changed: update test to match spec
8. Re-run tests until they pass
9. Stamp final test report
10. Update status

**Manual mode flow (default / flag off):**
- Same as current: write tests → run → present results → stamp only on pass

### `/status`
Show the current `AUTO_REVIEW_MODE` setting so users can see which mode they're in.

## How to toggle
- `/config auto-review on` — turns on auto-execute for review and test commands
- `/config auto-review off` — turns it back off (default)
- Or manually edit `.claude/settings/auto-review.json`

## Open questions
- Should auto-execute apply to *all* future commands, or just ones invoked after the flag is set?
- Should there be a dry-run mode that shows what *would* be fixed without actually changing code?
- How does this interact with `/improvement-loop`? Should that also auto-execute?
