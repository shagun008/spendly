# Improvement Log — 15.5

## Trigger Stage
`review` — After implementing the Release Notes Modal, the roadmap page showed grey dots for feature 16 (parent) and an incorrectly green Planned dot for 16.1 (child).

## Issue Summary
1. **Feature 16 parent row had grey dots** for Spec'd, Implemented, Tested, Reviewed — even though 16.1 (child) had all those timestamps set. The parent feature row didn't inherit pipeline timestamps from its children.
2. **Feature 16.1 child had Planned dot green** — `planned_at` was set on the child, but planning happens at the parent feature level, not individual releases.

## Root Cause
- When a release sub-row is shipped, its pipeline timestamps (`spec_at`, `implemented_at`, `tested_at`, `reviewed_at`) were not propagated up to the parent feature row. The roadmap template renders dots based on the parent's own timestamps.
- `planned_at` was incorrectly set on child rows (16.1) in the seed data. Planning is a parent-level activity.

## Fix Applied
1. **DB update:** Propagated child 16.1's pipeline timestamps to parent 16 using `COALESCE` to avoid overwriting existing values.
2. **DB update:** Set `planned_at = NULL` on 16.1 (child rows shouldn't have their own `planned_at`).
3. **Seed data update:** Updated `database/db.py` seed tuples for 16 and 16.1 to reflect the correct timestamps — parent has all pipeline stages, child has `planned_at=NULL`.

## Improvement Applied
- Parent feature rows now correctly show all green dots when their children have completed those stages.
- Child release rows no longer show a green Planned dot (planning belongs to parent).

## Architectural Notes
- The `get_all_features()` query returns a flat list. Parent-child relationships exist via `parent_number` but timestamps are stored per-row. When rendering the roadmap, the parent's own timestamps determine its dots — children's timestamps don't automatically propagate.
- **Future pattern:** When shipping a release, always propagate pipeline timestamps from child to parent using `COALESCE`. Also clear `planned_at` on child rows since planning is a parent-level activity.

## Future Prevention
- Add a rule to CLAUDE.md: "When shipping a release, propagate child pipeline timestamps to parent and clear child-only timestamps (`planned_at`, `captured_at`)."
- The `/ship-feature` command's Step 9a already has SQL for propagating timestamps — ensure it runs correctly for all parent features.
