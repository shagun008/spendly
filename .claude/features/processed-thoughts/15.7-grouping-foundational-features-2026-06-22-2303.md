# Processing Foundational Features

## Problem / Goal
The roadmap page shows all features/releases flat, making it hard to scan. Users need a cleaner hierarchical view where features 1–10 are grouped under a collapsible "Foundational Features" parent row, and parent features expand/collapse to show their child release rows instead of a description card.

## Who benefits
All users viewing the roadmap page (public page).

## Success looks like
Features 1–10 are hidden under a "Foundational Features" group row by default. Clicking expands/collapses the group. Parent feature chevrons now toggle child release rows instead of description cards. The description detail row is removed entirely.

## Constraints, risks, dependencies
No backend/data model changes needed — hierarchy already exists in the DB. Changes are limited to `templates/roadmap.html`, `static/css/roadmap.css`, and inline JS. Open design questions remain: (1) accordion vs multi-expand for parent features, (2) default state of the "Foundational Features" group on page load.

## Implementation ideas / open questions
- Add a `.roadmap-group-row` with chevron for "Foundational Features"
- Mark rows 01–10 with `data-parent-group="foundational"`
- Replace description-card expand/collapse with child-row toggling via `data-parent` attributes on release rows
- Remove deprecated `.roadmap-detail-row` / `.roadmap-detail-card` CSS
- No backend changes — `get_all_features()` already returns the hierarchy
