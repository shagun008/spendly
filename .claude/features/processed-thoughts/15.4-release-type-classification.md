# Processed Thought: Release-Level Type Classification

## Problem / Goal
The type column is set on parent feature rows during /capture-thoughts and hardcoded to 'feature', losing meaningful information. Type should be set per release sub-row during /plan-release when the work is actually understood.

## Who benefits
Developer/project owner — cleaner data model and more meaningful roadmap display.

## Success looks like
Parent rows have no type; each release sub-row has one of new-feature, enhancement, or bug-fix. Roadmap page renders the release type as a badge on sub-rows.

## Constraints, risks, dependencies
Low risk — type column schema unchanged, parent rows simply get NULL. Single-type per release only, no aggregation needed. Depends on 15.1 (features table exists) and 15.3 (harness commands already writing to DB).

## Implementation ideas / open questions
1. capture-thoughts.md — remove type question, drop type from parent DB insert.
2. plan-release.md — assign type per release, include in DB insert for sub-rows.
3. Roadmap page — render type badge on release rows.

## Release pressure / deadlines
None. Quality/accuracy improvement, not urgent.
